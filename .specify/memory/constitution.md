# vocastock Constitution

## Core Principles

### I. ドメインモデル先行と識別子命名
`docs/internal/domain/*.md` に定義された値オブジェクト、集約、ドメインイベント、
リポジトリ契約を実装の基準とする。これらに変更がある機能は、コード変更と同じ変更
セット内で対象文書を更新し、未反映のまま実装を先行させてはならない。識別子型は
必ず `XxxIdentifier` と命名し、`id`、`ID`、`xxxId` を使用してはならない。集約自身の
識別子フィールド名は常に `identifier` とし、他集約や他概念の識別子を保持する
フィールド名は `bankIdentifier` や `bankId` ではなく `bank` のように概念名そのものを
使わなければならない。理由: vocastock は学習概念の境界が多く、用語、不変条件、
識別子の役割がずれると仕様破綻と読解コスト増大を招くため。

### II. 非同期生成は完了結果のみ公開
解説生成と画像生成は、明示的なジョブ状態、再試行安全な冪等コマンド、失敗理由の追跡
を備えた非同期ワークフローとして設計する。ユーザーへ表示してよい生成物は完了済みの
結果のみであり、生成途中や不完全な内容を見せてはならない。理由: 学習体験では中間
生成物の誤表示が誤学習と信頼低下を招くため。

### III. 外部依存はポート越しに接続する
AI サービス、画像ストレージ、音声やメディア参照、単語検証などの外部依存は、
ドメイン層から直接呼び出さず、明示的なポートとアダプタ経由で接続する。ベンダー固有
SDK をドメインモデルへ持ち込まず、契約テストまたは統合テストで振る舞いを固定する。
理由: 生成品質、コスト、可用性の変動に対して差し替えと検証を可能にするため。

### IV. 仕様駆動で独立に届ける
仕様、計画、タスクはユーザーストーリー単位で独立に実装、検証、デモ可能でなければ
ならない。別ストーリーへの依存が避けられない場合は、計画書の Complexity Tracking
に理由と代替案を明記する。理由: 学習体験に関わる機能を小さく検証し、生成系の失敗
範囲を限定するため。

### V. 学習概念を混同しない
頻出度、知的度、習熟度、登録状態、解説生成状態、画像生成状態は別概念として扱い、
UI 文言、API、永続化、分析軸でも混同してはならない。名称変更や統合が必要な場合は、
ユーザー影響、移行方針、既存データの扱いを仕様に明記する。理由: vocastock の価値は
学習指標の分離に依存し、概念の混同は誤った評価や表示不整合を生むため。

## Product Constraints

- 英単語の登録可否、解説の有無、画像の有無は独立した状態として保持する。
- 生成ワークフローは少なくとも `pending`、`running`、`succeeded`、`failed` を
  区別できる設計にする。
- 画像などの生成成果物は永続化先で安定した識別子を持ち、再取得可能でなければならない。
- ユーザー向け画面は、生成中または失敗中の状態を表示してよいが、生成物本体は完了後に
  限って表示する。
- 新しい外部依存を追加する場合、タイムアウト、再試行、障害時の代替動作を計画書に記載
  する。

## Naming Constraints

- `id`、`ID`、`xxxId` は型名、フィールド名、契約名として使用してはならない。
- 識別子の型名は `XxxIdentifier` 形式に統一しなければならない。
- 集約自身を識別するフィールド名は常に `identifier` でなければならない。
- 集約が他集約または他概念の識別子を保持する場合、フィールド名は `xxxIdentifier` では
  なく `xxx` としなければならない。
- 例として、`User` が `BankIdentifier` を保持する場合、`User` 自身の識別子は
  `User.identifier`、銀行参照は `User.bank` と命名しなければならない。

## Workflow & Quality Gates

- `spec.md` では、独立に検証可能なユーザーストーリー、受け入れ条件、生成状態の
  ユーザー表示方針を定義する。
- `plan.md` の Constitution Check では、対象ドメイン文書、非同期ジョブの冪等性、
  外部依存のポート分離、概念境界の維持、識別子命名規約の遵守を確認する。
- `tasks.md` では、必要に応じてドメイン文書更新、ジョブ状態管理、契約テスト、
  統合テスト、表示制御、識別子命名統一のタスクを明示する。
- 実装レビューでは、ドメイン文書とコードの差分、未完了生成結果の非表示、外部依存の
  分離、ストーリー単位の独立検証可能性、`Identifier` 命名と関連参照フィールドの命名を
  確認する。
- 憲章に反する例外は一時対応として扱わず、仕様、計画、移行方針を伴う明示的な合意が
  ない限り許可しない。

## Governance

この憲章はリポジトリ内の他の運用文書より優先される。変更は理由、影響範囲、関連
テンプレート更新を含む pull request または同等のレビュー可能な変更として提案し、
承認後に反映する。バージョンはセマンティックバージョニングで管理し、原則の削除または
後方互換性のない再定義は MAJOR、新しい原則や実質的な運用追加は MINOR、語句修正や
明確化は PATCH とする。すべての計画レビューと実装レビューは、この憲章への適合確認を
必須とする。
