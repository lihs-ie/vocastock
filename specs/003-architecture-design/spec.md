# Feature Specification: アーキテクチャ設計

**Feature Branch**: `[003-architecture-design]`  
**Created**: 2026-04-15  
**Status**: Ready for Planning  
**Input**: User description: "アーキテクチャ設計を行う。私と議論しながら進める"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 全体責務を定義する (Priority: P1)

設計担当者として、プロダクト全体でどの責務をどの構成要素が担うかを一目で判断できるアーキテクチャを定義したい。これにより、新機能追加時に実装場所と依存方向を迷わず決められる。

**Why this priority**: 責務境界が曖昧なままだと、以後の設計・実装判断が一貫しないため。

**Independent Test**: 解説生成、画像生成、登録済み単語管理、結果表示の各責務について、第三者が担当境界を説明できれば独立に価値を確認できる。

**Acceptance Scenarios**:

1. **Given** 開発者が新機能の追加先を検討している, **When** アーキテクチャ設計書を読む, **Then** 各責務の所属境界と依存方向を判断できる
2. **Given** レビュー担当者が設計妥当性を確認している, **When** 構成要素間の責務一覧を見る, **Then** 重複責務や責務漏れを指摘できる

---

### User Story 2 - 非同期生成と表示規則を整理する (Priority: P2)

設計担当者として、解説生成と画像生成の非同期フロー、失敗時の扱い、ユーザーへの表示規則を一貫した形で整理したい。これにより、生成中や失敗時でも利用者体験を壊さない設計判断ができる。

**Why this priority**: このプロダクトの主要価値は非同期生成に依存しており、ここが曖昧だと実装の整合性が崩れるため。

**Independent Test**: 第三者が pending/running/succeeded/failed の各状態で「何を保存し、何を表示し、誰が再試行を担うか」を説明できれば独立に価値を確認できる。

**Acceptance Scenarios**:

1. **Given** 解説生成または画像生成が進行中である, **When** アーキテクチャ設計書を参照する, **Then** 保存状態、再試行責務、ユーザー表示方針を判断できる
2. **Given** 外部生成サービスが失敗する, **When** エラー時の責務分担を確認する, **Then** 中間生成結果を表示せずに復旧方針を説明できる

---

### User Story 3 - 外部境界と運用境界を合意する (Priority: P3)

設計担当者として、AI・ストレージ・検証などの外部依存と、各構成要素の運用境界を定義したい。これにより、将来の差し替えや運用変更を影響範囲付きで議論できる。

**Why this priority**: 外部依存の境界が曖昧だと、ベンダー変更や運用改善時に影響が読めないため。

**Independent Test**: 第三者が外部依存ごとに「どこから呼ばれ、何を返し、失敗時にどこまで波及するか」を説明できれば独立に価値を確認できる。

**Acceptance Scenarios**:

1. **Given** 外部サービスの差し替えを検討している, **When** アーキテクチャ設計書を読む, **Then** 差し替え対象の境界と影響範囲を説明できる
2. **Given** 新しい運用要件が追加される, **When** 構成要素の運用境界を確認する, **Then** どの境界を変更すべきか判断できる

### Edge Cases

- 一つの機能要求が複数の境界にまたがる場合、主責務をどこに置くか
- 外部生成サービスが部分的な結果を返した場合でも、完了結果以外をユーザーに見せないルールをどう維持するか
- 既存ドメイン文書の責務定義と新しいアーキテクチャ判断が衝突した場合、どの文書を先に更新するか

## Domain & Async Impact *(mandatory when applicable)*

- **Domain Models Affected**: `docs/internal/domain/common.md`, `docs/internal/domain/explanation.md`, `docs/internal/domain/visual.md`, `docs/internal/domain/service.md`
- **Invariants / Terminology**: frequency、sophistication、proficiency、登録状態、解説生成状態、画像生成状態は別概念として保ち、責務境界でも混同しない
- **Async Lifecycle**: explanation と visual image は pending/running/succeeded/failed を区別し、再試行は冪等に扱う。途中成果物は内部状態としてのみ扱う
- **User Visibility Rule**: ユーザーには完了した生成結果のみを表示し、pending/failed は状態表示のみに留める
- **Identifier Naming Rule**: 識別子型は `XxxIdentifier`、集約自身は `identifier`、関連識別子は概念名で表現し、`id` / `xxxId` / `xxxIdentifier` を使わない
- **External Ports / Adapters**: 英単語検証、解説生成、画像生成、画像永続化、結果取得の各境界を整理対象にする

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: アーキテクチャ設計は、Flutter クライアント、非同期生成基盤、保存、結果表示、外部依存との接続を含む end-to-end 全体を対象として責務境界を定義しなければならない
- **FR-002**: 設計は、英単語登録、解説生成、画像生成、結果表示の各責務について、主責務を持つ境界を一意に示さなければならない
- **FR-003**: 設計は、解説生成と画像生成の非同期ライフサイクル、再試行責務、失敗時の扱いを定義しなければならない
- **FR-004**: 設計は、ユーザーに完了済み結果のみを見せる表示規則を、状態遷移と整合する形で定義しなければならない
- **FR-005**: 設計は、英単語検証、生成、画像保存などの外部依存について、呼び出し境界と責務分担を定義しなければならない
- **FR-006**: 設計は、構成要素間の境界を運用単位と主要データフローまで表現し、主要な同期処理・非同期処理・失敗時分岐の流れをレビュー可能な形で示さなければならない
- **FR-007**: 設計は、既存ドメイン文書と整合する用語集と不変条件を含まなければならない
- **FR-008**: 設計は、各新規要件をどの境界に割り当てるかをレビュー可能な形で示さなければならない
- **FR-009**: 設計は、将来の外部サービス差し替えや運用変更時に影響範囲を判断できる境界情報を含まなければならない
- **FR-010**: 設計成果物は、理想形の target architecture に加えて、現状からそこへ到達するための段階移行方針を明示しなければならない

### Key Entities *(include if feature involves data)*

- **Architecture Boundary**: どの責務を持ち、どこまで依存してよいかを表す単位
- **Async Flow Contract**: 解説生成・画像生成における状態、入力、完了条件、失敗時の扱いを表す定義
- **External Port Contract**: 外部依存との入出力責務と失敗時境界を表す定義
- **Architecture Decision**: なぜその責務分担と境界を採用するかを示す判断記録

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: レビュー参加者が 10 分以内に、単語登録・解説生成・画像生成・結果表示の各責務を 1 つの主境界へ対応付けられる
- **SC-002**: 外部依存を伴う主要フローの 100% について、呼び出し境界、失敗時責務、表示規則の整合を説明できる
- **SC-003**: 解説生成と画像生成の各状態について、ユーザーに見える状態と内部状態の違いを 100% 定義できる
- **SC-004**: 憲章と既存ドメイン文書に対する矛盾がレビュー時に 0 件である

## Assumptions

- 今回の feature は実装そのものではなく、後続の plan と implementation に使う設計成果物を主対象とする
- 既存の憲章、用語、不変条件は有効であり、アーキテクチャ設計はそれらを上書きせず整合させる
- 現時点では英単語登録、解説生成、画像生成、結果表示がコア業務フローである
- 今回の設計はエンドツーエンド全体を対象とし、運用単位と主要データフローまで表現する
- 設計成果物には理想形だけでなく、現状からの段階移行方針も含める
