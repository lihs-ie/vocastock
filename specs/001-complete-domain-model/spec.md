# Feature Specification: ドメインモデル設計書の完成

**Feature Branch**: `001-complete-domain-model`  
**Created**: 2026-04-14  
**Status**: Draft  
**Input**: User description: "ドメインモデル設計書を完成させる"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 核心概念を定義する (Priority: P1)

ドメイン設計者として、英単語登録、解説生成、画像生成、学習状態管理に必要な核心概念が
漏れなく定義された設計書を持ちたい。そうすることで、実装前に語彙、責務、不変条件の
解釈を統一できる。

**Why this priority**: 核心概念が曖昧なままでは、以後の設計、実装、レビューの判断基準が
ぶれるため。

**Independent Test**: 設計書だけを読んで、解説、画像、学習状態に関する主要概念の意味、
保持項目、関係、不変条件を第三者が説明できれば成立する。

**Acceptance Scenarios**:

1. **Given** 既存のドメイン文書に未定義や不足がある, **When** 設計書完成後に確認する,
   **Then** 主要な値オブジェクト、集約、ドメインサービス、イベント、リポジトリ責務が
   一貫した用語で定義されている
2. **Given** 新しい実装担当者が設計書を読む, **When** 解説や画像に関する振る舞いを追う,
   **Then** 追加説明なしで概念の境界と責務を理解できる

---

### User Story 2 - 非同期生成の業務意味を明確にする (Priority: P2)

実装担当者として、解説生成と画像生成の状態遷移、再生成条件、ユーザー表示ルールが
設計書に明記されていてほしい。そうすることで、完了結果のみを表示する要件を崩さずに
実装できる。

**Why this priority**: 非同期生成の定義が曖昧だと、中間結果の誤表示や再試行時の不整合が
発生しやすいため。

**Independent Test**: 設計書を読んだレビュー担当者が、生成依頼から完了・失敗・再生成まで
の業務状態と表示可否を説明できれば成立する。

**Acceptance Scenarios**:

1. **Given** 解説生成または画像生成を扱う機能の設計レビューを行う, **When** 設計書を確認する,
   **Then** 状態、遷移条件、再試行の扱い、ユーザーへ見せてよい情報の範囲が定義されている
2. **Given** ユーザー表示ルールを確認する, **When** 生成途中または失敗時の扱いを確認する,
   **Then** 中間生成結果は表示せず、状態表示のみ許可されることが明記されている

---

### User Story 3 - 文書間の整合性を担保する (Priority: P3)

レビュー担当者として、要件、ADR、ドメインモデル設計書の間に矛盾がない状態にしたい。
そうすることで、後続の仕様化や計画策定で前提の食い違いを減らせる。

**Why this priority**: ドキュメント間の不整合は、実装着手後に手戻りを生むため。

**Independent Test**: 要件文書、ADR、ドメイン文書を横断確認し、主要用語と業務ルールに
矛盾がないことを第三者が判定できれば成立する。

**Acceptance Scenarios**:

1. **Given** 要件文書とドメイン文書を比較する, **When** 頻出度、習熟度、生成状態などの用語を確認する,
   **Then** 用語の意味と区別が一貫している
2. **Given** ADR とドメイン文書を比較する, **When** 外部サービスとの責務分担を確認する,
   **Then** どの役割をドメイン外へ委譲するかが矛盾なく説明されている

---

### Edge Cases

- 既存文書内で、画像を `Explanation` の値として持つ定義と `VisualImage` 集約として扱う定義が競合する場合
- 英単語の登録状態が要件上は必要でも、ドメイン文書内に独立した概念として未定義のまま残る場合
- 解説生成状態と画像生成状態が混同され、どの結果をユーザーに見せてよいか判断できない場合
- 外部 AI、単語存在確認、ストレージの責務境界が曖昧で、ドメイン責務と混線する場合

## Domain & Async Impact *(mandatory when applicable)*

- **Domain Models Affected**: `docs/internal/domain/common.md`, `docs/internal/domain/explanation.md`,
  `docs/internal/domain/service.md`, `docs/internal/domain/visual.md`
- **Invariants / Terminology**: 頻出度、知的度、習熟度、登録状態、解説生成状態、
  画像生成状態を別概念として定義し、それぞれの意味、用途、相互関係を明記する
- **Async Lifecycle**: 解説生成と画像生成について、少なくとも `pending`、`running`、
  `succeeded`、`failed` の業務状態、再試行、再生成、重複依頼時の扱いを文書化する
- **User Visibility Rule**: ユーザーに表示してよい生成物は完了済み結果のみとし、
  生成中・失敗時は状態のみを表示対象とする
- **Identifier Naming Rule**: 識別子型は `XxxIdentifier` とし、集約自身の識別子フィールドは
  `identifier`、関連識別子フィールドは `entry` や `image` のように概念名で定義する
- **External Ports / Adapters**: 英単語存在確認、登録済み判定、解説生成、画像生成、
  生成画像の永続化、外部発音リソース参照の責務をドメイン外インターフェースとして整理する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 仕様は、英単語登録、解説生成、画像生成、学習状態管理に必要な主要ドメイン概念を漏れなく定義しなければならない
- **FR-002**: 仕様は、対象となる値オブジェクト、集約、ドメインサービス、ドメインイベント、リポジトリ責務について、意味、保持情報、関係、不変条件、業務上の振る舞いを定義しなければならない
- **FR-003**: 仕様は、頻出度、知的度、習熟度、登録状態、解説生成状態、画像生成状態を別概念として区別し、その違いを明文化しなければならない
- **FR-004**: 仕様は、解説生成と画像生成の依頼、進行、完了、失敗、再試行、再生成の業務ルールを定義しなければならない
- **FR-005**: 仕様は、ユーザーに表示される情報と内部状態を区別し、生成途中または失敗中の中間生成結果を表示しないルールを定義しなければならない
- **FR-006**: 仕様は、外部 AI、単語検証、ストレージなどの外部責務を、ドメインの概念と区別された役割として定義しなければならない
- **FR-007**: 仕様は、既存の要件文書と ADR と整合する用語を採用し、差異がある場合はどの文書を正とするかを明示しなければならない
- **FR-008**: 仕様は、今回の完成対象としない論点や前提条件を明示し、設計書の利用者が誤って実装範囲を広げないようにしなければならない
- **FR-009**: 仕様は、識別子型を `XxxIdentifier` で統一し、集約自身の識別子以外を保持する
  フィールド名では `id`、`xxxId`、`xxxIdentifier` を使わない命名規約を定義しなければならない

### Key Entities *(include if feature involves data)*

- **Explanation**: 英単語の解説を表す中心集約であり、意味、発音、頻出度、知的度、例文、類似表現、語源、画像との関係を持つ
- **VisualImage**: 英単語を視覚化した結果を表す概念であり、解説との対応、識別方法、再生成時の扱いを持つ
- **Generation State**: 解説生成および画像生成の依頼から完了までの状態を表す業務概念であり、表示可否や再試行条件の判断基準となる
- **Learning Indicators**: 頻出度、知的度、習熟度など、英単語学習上の別々の評価軸を表す概念群

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 主要ドメイン文書の 100% が、対象概念の意味、保持情報、関係、不変条件、振る舞いのうち必要な項目を欠落なく記載している
- **SC-002**: 新しい実装担当者またはレビュー担当者が、設計書のみを用いて 5 分以内に主要概念とその境界を説明できる
- **SC-003**: 要件文書、ADR、ドメイン文書を横断確認した際に、頻出度、知的度、習熟度、登録状態、解説生成状態、画像生成状態に関する矛盾が 0 件である
- **SC-004**: 解説生成と画像生成について、ユーザーに見せる情報の範囲と業務状態の対応を、レビュー担当者が曖昧さなく判定できる

## Assumptions

- 今回の対象はドメインモデル設計書と関連文書の整備であり、アプリケーションコードの実装変更は含まない
- 要件文書と ADR は現時点のプロダクト意図を表す基準文書であり、整合のためにドメイン文書側を主に補完する
- 認証、権限管理、複数ユーザー間の共有などの周辺業務は、この feature の主対象外とする
- 外部 AI、ストレージ、メディア配信の具体的な製品選定は行わず、業務上必要な役割と責務のみを定義対象とする
