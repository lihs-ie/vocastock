## User Scenarios & Testing *(mandatory)*

### User Story 1 - ローカル環境を再現する (Priority: P1)

アプリ開発者として、Flutter 開発に必要なローカル環境を短時間で再現したい。そうすることで、
個人差の少ない手順で開発を開始でき、Firebase を利用する機能もローカルで検証できる。

**Why this priority**: ローカル環境の再現性がないと、実装前の立ち上がりと不具合切り分けが
継続的に遅くなるため。

**Independent Test**: 新規参加者が案内された手順だけで必要なツールを準備し、アプリの起動、
ローカルエミュレーション環境への接続、基本的な検証を完了できれば成立する。

**Acceptance Scenarios**:

1. **Given** 開発者が未セットアップの端末を持っている, **When** 開発環境手順に従う,
   **Then** Flutter 開発に必要なローカルツール、コンテナ化されたローカルエミュレーション、
   補助ツールが揃い、開発を開始できる
2. **Given** 開発者が Firebase を利用する機能を確認したい, **When** ローカル環境を起動する,
   **Then** 共有のクラウド環境に依存せず、ローカルで検証できる

---

### User Story 2 - CI で品質を確認する (Priority: P2)

レビュー担当者として、ブランチ更新ごとに必要な検証が自動で走る状態にしたい。そうすることで、
壊れた依存関係や不十分な変更が保護対象ブランチへ入るのを防げる。

**Why this priority**: ローカル差分だけに依存すると、レビュー前後で再現不能な失敗が増えるため。

**Independent Test**: 任意の変更ブランチに対して自動検証が実行され、必要な検証結果が揃わない
限り保護対象ブランチに統合できないことを確認できれば成立する。

**Acceptance Scenarios**:

1. **Given** 開発者が変更をブランチへ反映する, **When** 自動検証が開始される,
   **Then** 環境整合性確認、静的検査、テスト、最低限のビルド検証が実行される
2. **Given** 必須検証のいずれかが失敗する, **When** 統合判定を確認する,
   **Then** 保護対象ブランチへの統合は許可されない

---

### User Story 3 - 採用バージョンの安全性を説明する (Priority: P3)

保守担当者として、開発環境と CI で採用する各コンポーネントのバージョン選定理由と、
最新実機で検証済みの承認済み baseline を説明できるようにしたい。そうすることで、
将来の更新判断や脆弱性対応を属人化させずに済む。

**Why this priority**: セキュリティと保守性の根拠が曖昧だと、環境更新が後手に回りやすいため。

**Independent Test**: 採用対象ごとに、サポート状況、LTS または安定版採用理由、既知脆弱性の
調査結果、更新判断ルール、および最新実機との差分反映方針を第三者が確認できれば成立する。

**Acceptance Scenarios**:

1. **Given** 採用バージョン一覧を確認する, **When** 各コンポーネントの選定根拠を見る,
   **Then** 公式サポート状況、脆弱性調査結果、採用理由が記録されている
2. **Given** あるコンポーネントに正式な LTS が存在しない, **When** 採用方針を確認する,
   **Then** 公式にサポートされている安定系統を採用対象とし、その理由が記録されている
3. **Given** 実機の host toolchain が既存の承認済み baseline より新しくなった, **When** version governance を確認する,
   **Then** 実機で確認済みの新 baseline、採用理由、見直し条件、関連文書と検証基準の更新対象が記録されている

---

### Edge Cases

- ローカル端末に必要なコンテナ実行基盤がない、または利用できない場合
- ローカルエミュレーションに必要な補助ツールは揃っているが、Flutter 実行環境だけが不足している場合
- 採用予定コンポーネントに正式な LTS 提供がなく、安定版の採用判断が必要な場合
- 採用後に medium severity 以上の既知脆弱性が判明し、更新または回避策の判断が必要になる場合
- ブランチは通るが保護対象ブランチへの統合時に再実行が必要な場合
- 最新実機の host toolchain が既存の承認済み exact version より新しく、ダウングレードより baseline 更新の方が現実的な場合

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、Flutter 開発に必要なローカルツール、コンテナ化されたローカル Firebase エミュレーション環境、その他必須補助ツールの一覧と準備手順を定義しなければならない
- **FR-002**: システムは、未セットアップの端末から開発開始可能な状態までの再現可能なローカル構築手順を提供しなければならない
- **FR-003**: システムは、共有クラウド環境に依存せずに主要な開発検証を行えるローカル Firebase エミュレーション手段を提供し、プロジェクトで利用する全 Firebase サービスをローカルで検証可能にしなければならない
- **FR-004**: システムは、iOS / Android / macOS を対象プラットフォームとして扱い、公式サポートするローカル開発ホストを macOS と定めた上で、ローカル構築時に必要な追加前提条件、対応ホスト環境、既知の制約を明示しなければならない
- **FR-005**: システムは、各採用コンポーネントについて、正式な LTS がある場合はサポート中の LTS を採用対象とし、正式な LTS がない場合はサポート中の安定系統を採用対象とする方針を定義しなければならない
- **FR-006**: システムは、各採用バージョンについて、公式サポート状況、既知脆弱性の調査結果、採用理由、見直し条件を記録しなければならない。実機で検証済みの host toolchain が新 baseline になる場合は、その差分と更新理由も同じ変更単位で記録しなければならない
- **FR-007**: システムは、変更ブランチと `main`、`develop`、`release/*` への統合判定に必要な自動検証を定義しなければならない
- **FR-008**: システムは、自動検証に少なくとも環境整合性確認、静的検査、テスト、最低限のビルド検証を含めなければならない
- **FR-009**: システムは、必須検証が失敗した変更を保護対象ブランチへ統合できない運用ルールを定義しなければならない
- **FR-010**: システムは、依存関係または採用コンポーネントに対する脆弱性検査で `Critical`、`High`、`Medium` のいずれかが検出された場合、統合判定をブロックしなければならない
- **FR-011**: システムは、ローカル環境と CI の双方で利用する設定情報の管理方針を定義し、秘匿すべき情報とローカル既定値を区別しなければならない
- **FR-012**: システムは、環境障害やバージョン更新時の切り分けに必要なトラブルシューティング導線を提供しなければならない

### Key Entities *(include if feature involves data)*

- **Local Environment Profile**: Flutter 開発に必要なローカル構成要素、前提条件、公式サポートする macOS ホスト条件、およびプロジェクトで使う全 Firebase サービスのローカル再現範囲を表す定義
- **Target Platform Matrix**: iOS / Android / macOS それぞれに必要なローカル要件と CI 要件の対応関係を表す定義
- **Version Approval Record**: 採用コンポーネントごとのサポート状況、脆弱性調査、採用理由、見直し条件を表す記録
- **CI Check Suite**: ブランチ更新や統合判定時に必要な自動検証の集合
- **Branch Policy**: どのブランチにどの検証を要求するか、統合可否をどう決めるかを表す運用ルール
- **Protected Branch Set**: `main`、`develop`、`release/*` に対する必須検証と統合条件を表す定義

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新規開発者が案内された手順のみで 60 分以内にローカル開発環境を構築し、基本的な開発開始確認を完了できる
- **SC-002**: 保護対象ブランチへ入るすべての変更が、定義済み必須 CI 検証を 100% 通過してから統合される
- **SC-003**: 採用対象コンポーネントの 100% に対して、サポート状況、脆弱性調査結果、採用理由、見直し条件が記録される
- **SC-004**: 採用時点で、medium severity 以上の既知脆弱性が未評価のまま残る採用コンポーネントが 0 件である

## Assumptions

- Flutter は開発対象として固定であり、今回の feature はアプリ実装ではなく開発基盤整備を対象とする
- 対象プラットフォームは iOS / Android / macOS とする
- 公式サポートするローカル開発ホスト OS は macOS のみとする
- Flutter 自体に正式な LTS 呼称がない場合は、公式にサポートされている安定系統を採用対象とし、その判断根拠を記録する
- Firebase エミュレーターの対象は、プロジェクトで利用する全 Firebase サービスとする
- 承認済み host toolchain baseline は、公式サポート状況と脆弱性調査を満たしたうえで、最新実機で検証済みの macOS baseline に合わせて更新する
- 現時点の実機 baseline は macOS 26.4.1、Flutter 3.41.5、Xcode 26.4、Android Studio 2025.3、CocoaPods 1.16.2、Docker Desktop 4.69.0 とする
- 初期スコープでは、日常開発に必要なローカル環境、ローカル Firebase エミュレーション、継続的検証、統合前検証を優先し、配布ストア公開手順や本番リリース運用は含めない
- ブランチ運用は、短命な作業ブランチと `main`、`develop`、`release/*` を保護対象とする運用を前提とする
